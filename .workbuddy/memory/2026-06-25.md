## 排弹系统架构产品视角评估

### 背景
用户质疑 DDR-004/005 的架构修改"是不是只为了开发服务"，要求从产品视角全面评估架构。

### 架构师产品视角评估
- 重新阅读 SRS 和 SDD，从核心价值、质量属性、演进能力、约束满足四个维度评估
- 发现 5 个架构缺失盲区：
  1. 安全架构完全空白（致命）——无紧急停止、无安全联锁、无 fail-safe
  2. 实时数据管线有根本缺陷——MQTT 无法支撑 128MB/s 点云数据，SRS I-002 明确写了"共享内存"
  3. 离线模式无设计——SRS 要求离线工作≥2小时，架构无任何模块负责
  4. 可靠性架构缺失——无 7×24 守护、健康监控、崩溃恢复
  5. 殉爆安全链无架构隔离——只是 PathPlanner 的一个方法

### 架构师诚实承认
- DDR-001~005 把"架构"等同于"代码结构"，忽略了"运行时行为设计"
- 产品架构要回答：安全怎么保证？性能怎么达标？7×24 怎么实现？离线怎么工作？

### 决策
- Phase 1 新增 5 项架构工作：
  - A. 安全架构骨架（P0）：SafetyManager + EmergencyStopController + SafetyInterlock + FailSafeHandler
  - B. 分层数据通道（P0）：共享内存/ZeroMQ 传高带宽 + MQTT 传控制遥测 + Qt 信号槽传 UI 更新
  - C. 离线模式骨架（P1）：OfflineManager + OfflineDataCache + CommandQueue
  - D. 可靠性骨架（P1）：HealthMonitor + Watchdog + LogManager
  - E. 殉爆安全链独立模块（P1预研）：SafetyChainModule，独立编译独立测试
- Phase 1 从 10 改造项扩展到 15 项
- 已落盘 DDR-006

### 关键矛盾发现
- MQTT 作为唯一数据通道 vs SRS 高带宽实时数据需求
- FrontendDataStore 进程内单例 vs SRS 多席位协同（≥10人）需求

## 代码架构之道（DDR-007）

### 背景
用户提出"除了术之外，还可以考虑代码层面的道"——要求确立架构哲学/架构原则，不是技术选型。

### 架构师深度思考
- **第一性原理：安全独立性**——安全是系统存在的必要条件，速度是充分条件
  - 安全不在同一维度上排序，它是"地面"，其他是"地上建筑"
  - 唯一"违背了就不可逆"的原理（速度可等、可靠可修、一致可同步，但爆炸不可回滚）
  
- **5 条架构原则**：
  1. 安全链物理隔离——进程级隔离，不是类级隔离。业务进程崩溃时安全进程仍在运行
  2. 故障即安全——任何故障的默认结果是安全态，不是继续操作或未定义状态
  3. 数据通道分级——高带宽(共享内存/ZeroMQ) + 控制遥测(MQTT) + UI更新(Qt信号槽)，不按方便统一
  4. 外部依赖零信任——假设所有外部依赖失效，为每个关键功能定义降级路径
  5. 安全路径可独立验证——安全链可独立编译/运行/测试，不依赖业务模块

### 用原则审视 DDR-001~006
- 最严重违背：6 份 DDR 没有任何一份提过"紧急停止"三个字
- DDR-003 全走 MQTT 违背原则 3
- DDR-005 QVariantMap extensions 违背原则 5，正式废除
- DDR-001~005 全在回答"代码结构"问题，没回答"运行时行为"问题

### 如果重来的顺序
- 先立"道"（架构原则）→ 再定"法"（安全/数据流/可靠性架构）→ 最后行"术"（代码组织/并行策略）
- 已落盘 DDR-007
