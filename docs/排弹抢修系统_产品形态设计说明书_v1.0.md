## 1. 设计思路与总体理念

### 1.1 核心设计目标

本系统的本质问题是：**敌方打击后，如何在最短时间内让一条3000m×500m的跑道恢复起降能力？**

这个问题可以拆解为四个子问题：

1. **找到所有未爆弹**——空中快速普查 + 地下精确复察
2. **判明威胁等级**——多传感器融合识别 + AI分类
3. **安全排除威胁**——远程/无人化处置
4. **评估修复条件**——弹坑三维建模 + 跑道损毁量化

### 1.2 设计原则

| 原则 | 说明 | 来源 |
|------|------|------|
| **分层探测** | 空中快速普查（分钟级）→ 地面精确复察（仅针对可疑点），而非全场地毯式地面扫描 | USAF RADAS实践：sUAS先飞一遍，地面团队只去标注点 |
| **载荷集成** | 一次飞行同时采集LiDAR+红外+可见光，不拆分为多次飞行 | 论文结论：多传感器融合＞单一传感器；排弹方案也设计了集成吊舱 |
| **统一消息总线** | 所有设备通过MQTT + 指挥平台通信，不搞点对点直连 | 排弹方案系统架构：C/S + 消息队列 |
| **Mesh中继** | 机器人进入跑道凹坑/建筑物背后时，通过自组网中继保持链路 | USAF MDCR实践：Babel协议mesh网络用于EOD机器人 |
| **离线容灾** | 通信中断时设备本机可继续执行预设任务 | 排弹方案明确要求 |

## 2. 系统工作流程与设备角色

### 2.1 五阶段闭环流程

```
阶段1: 机场三维环境建模（战前/部署时一次性完成）
  │   执行设备: 2架无人机 + 集成吊舱
  │   产出: 机场厘米级数字孪生模型
  │   耗时: ~2小时（首次完整扫描）
  ▼
阶段2: 实时探测（遭受攻击后立即启动）
  │   执行设备: 2架无人机同时起飞
  │   任务: 对跑道全域普查，识别弹坑、地表未爆弹、地下可疑点
  │   耗时: ~30分钟（双机分区扫描）
  ▼
阶段3: 目标识别与态势融合（探测数据回传后自动执行）
  │   执行设备: 数据处理服务器（AI推理）
  │   任务: 多源数据融合 → 目标识别分类 → 三维标注 → 推送指挥席
  │   耗时: 实时（边采边算）
  ▼
阶段4: 排弹抢修（指挥员确认目标后）
  │   执行设备: 探测载车（可疑点复察） + 排爆机器人（处置）
  │   任务: GPR+磁力仪精确复察地下目标 → 机器人抵近处置
  │   耗时: 取决于目标数量和类型
  ▼
阶段5: 效果评估（处置后）
  │   执行设备: 1架无人机（复察飞行）
  │   任务: 处置区域重新扫描，验证弹药已排除，弹坑状态可修复
  │   耗时: ~15分钟
  ▼
 闭环: 评估数据回写数字孪生模型，更新跑道可用状态
```

### 2.2 每个设备在流程中具体做什么

| 阶段 | 设备 | 做什么 | 怎么做的 |
|------|------|--------|---------|
| 阶段1/2 | 无人机×2 | 对跑道进行网格化扫描飞行 | 两机分左右半区并行飞行，航线自动生成，飞行高度50m，速度10m/s |
| 阶段1/2 | 集成吊舱(LiDAR) | 获取跑道表面三维点云 | 从50m高度发射905nm激光，240k点/秒，精度2cm@150m |
| 阶段1/2 | 集成吊舱(红外) | 识别热异常目标 | 1280×1024分辨率，30mK灵敏度，自动标注温差＞2°C的区域 |
| 阶段1/2 | 集成吊舱(广角相机) | 拍摄跑道表面高清图片 | 2000万像素，与点云同步触发，用于AI形态识别 |
| 阶段3 | 数据处理服务器 | 融合三路传感器数据，运行AI推理 | 时间戳对齐→点云配准→热图上色→目标检测→输出结构化目标列表 |
| 阶段4 | 探测载车 | 对无人机标记的可疑地下点逐一复察 | 人工驾驶到标记点，GPR+磁力仪精确确认深度/尺寸/材质 |
| 阶段4 | 排爆机器人 | 抵近未爆弹，执行排除 | 远程遥控行驶到目标，根据AI推荐选择处置工具并操作 |
| 阶段5 | 无人机(复察) | 扫描处置后的区域 | 同阶段2流程，对比处置前后点云和热图 |

## 3. 物理架构：什么东西装在什么东西上

### 3.1 无人机 + 集成吊舱（空中平台）

这是整个系统最核心的设计决策。我们不做"分次挂载不同载荷"，而是一次飞行同时采集全部数据。

**设计依据**：
- 排弹方案0919列出的无人机载荷包括LiDAR、红外、广角相机三类传感器
- 学术文献综述结论：多传感器融合的探测效果优于任何单一传感器
- 机场跑道是平坦开阔区域，三类传感器的工作条件不冲突（不需要因红外需要低空而LiDAR需要高空之类的问题）

**吊舱内部接口**：

| 连接 | 接口类型 | 协议 | 说明 |
|------|---------|------|------|
| LiDAR → 主控 | 以太网(RJ45/板间) | TCP/IP | 点云数据流 + 控制指令 |
| 红外 → 主控 | 以太网(RJ45/板间) | RTSP | 红外视频流 + 控制指令 |
| 广角相机 → 主控 | USB 3.0 / 以太网 | RTSP / UVC | 可见光视频流 |
| 主控 → GNSS | 串口 + PPS | NMEA + 脉冲 | 统一时间戳 |

| 主控 → 无人机 | E-Port V2 (×1) | DJI PSDK协议 | 供电 + 数据传输 |
| 无人机 → 地面 | 5.8GHz图传 + 2.4GHz数传 | O3/O4协议 | 回传所有数据到地面站 |

**吊舱和无人机之间的电气接口**：

吊舱通过无人机的一个E-Port V2接口（或航空插头）获取：
- **供电**：DC 24V，峰值功耗 ≤80W（三个传感器+主控的总和）
- **数据**：千兆以太网，通过无人机数传链路回传
- **时间同步**：PPS脉冲信号

### 3.2 无人机机巢（地面保障）

每架无人机配一个机巢，机巢做什么：

| 功能 | 实现方式 |
|------|---------|
| 自动起降平台 | 电动开合顶盖 + 降落引导标识 |
| 自动充电 | 触点式充电，无人机降落即对接，充电功率 ≥500W |

| 远程控制 | 以太网连接至核心交换机，通过MQTT收发指令 |

| 连接 | 接口 | 说明 |
|------|------|------|
| 机巢 → 核心交换机 | 室外型超五类网线（铠装） | 控制指令+状态上报 |
| 机巢 → 市电 | 铠装电力电缆 YJV 3×4mm² | 220V/50Hz，峰值3kW |

### 3.3 探测载车（地面复察平台）

探测载车不是特制车辆，而是一套**可搭载于普通皮卡/越野车上的探测套件**。只在阶段4使用——无人机标记出地下可疑点后，载车才出动逐个复查。

**为什么是车载而不是机载**：
- 探地雷达（GPR）需要紧贴地面或非常近距离（天线距地面≤30cm）才有足够的信号穿透力
- 机载GPR目前仍处于研究阶段，实际应用非常有限（论文中仅占极少数实验）
- 磁力仪虽然可以机载（航空磁力仪是成熟技术），但精度不如地面近距离测量
- 跑道场景3000m×500m，如果用GPR全场扫描，耗时会达到数百小时——不现实
- 正确的策略：无人机快速普查标记可疑点 → 载车只去那几个点精确复察

          │   │ 通信模块             │             │
          │   │ Mesh中继节点(机载)   │             │
          │   │ + 4G/5G 备份        │             │
          │   │ + 有线光纤口(精操用) │             │
          │   └─────────────────────┘             │
          └─────────────────────────────────────┘
```

**机器人工具快换系统**：

排爆方案提到机器人"18种扩展工具"，实际核心场景只需要4种：

| 工具 | 用途 | 对应弹药类型 | 搭载方式 |
|------|------|-------------|---------|
| **夹爪** | 抓取、转移未爆弹 | 所有地表弹药 | 机械臂末端快换 |
| **定向能量武器** | 烧毁电子引信、远程引爆 | 电子引信（反跑道炸弹、封锁弹） | 机械臂末端快换 |
| **液压剪/扳手** | 缓冲拆解机械引信 | 机械引信（老式航弹） | 机械臂末端快换 |
| **切割器** | 切割弹体外壳、分离引信 | 半埋/裸露弹药 | 机械臂末端快换 |

**工具存储**：机器人底盘上设工具架，携带全部4种工具。切换时机器人自行回架换刀，无需人工干预。

## 4. 通信架构：谁和谁通话，怎么通

### 4.1 通信总体拓扑

```
                    ┌──────────────────┐
                    │   MQTT Broker     │  部署于数据处理服务器
                    │   (Mosquitto)     │  端口1883
                    │   核心消息中枢     │  所有设备通过它通信
                    └────────┬─────────┘
                             │
        ┌────────┬───────────┼───────────┬──────────┐
        │        │           │           │          │
    ┌───┴──┐ ┌───┴───┐ ┌───┴───┐ ┌────┴────┐ ┌──┴──────┐
    │指挥席 │ │服务器 │ │无人机 │ │探测载车 │ │排爆     │
    │工作站 │ │(本地)  │ │地面站 │ │工控机   │ │机器人  │
    └──────┘ └──────┘ └──┬──┬─┘ └────────┘ └──┬─────┘
                         │  │                  │
                    ┌────┘  └────────┐   ┌────┘
                    │               │   │
               ┌────┴────┐    ┌─────┴───┴──┐
               │无人机1   │    │Mesh中继节点 │  (部署于跑道边缘)
               │(机载电台)│    │(自组网)     │
               └─────────┘    └────────────┘
```

### 4.2 各设备通信方式详解

#### 4.2.1 指挥席工作站 ↔ 服务器

| 项目 | 详情 |
|------|------|
| 物理介质 | 光纤（千兆），经核心交换机 |
| 协议 | WebSocket（态势推送） + HTTP REST（查询/指令） |
| 数据内容 | 接收: 三维态势数据、目标列表、设备状态、告警；发送: 任务规划指令、设备控制指令 |
| 带宽 | ≥100Mbps |
| 延迟 | ≤50ms |
| 冗余 | 双光纤链路，断一自动切 |

#### 4.2.2 无人机 ↔ 地面站 ↔ 服务器

这是最关键的一段。无人机在空中采集三种传感器的海量数据，需要可靠回传。

| 链路 | 物理介质 | 协议 | 数据内容 | 带宽需求 | 作用距离 |
|------|---------|------|---------|---------|---------|
| **图传链路** | 5.8GHz无线 | RTSP / 私有协议 | 红外视频 + 可见光视频 | ≥20Mbps（双路1080p） | ≥5km |
| **数传链路** | 2.4GHz无线 | MQTT over TCP | 飞控遥测 + LiDAR点云（压缩）+ 控制指令 | ≥10Mbps | ≥5km |
| **地面站→服务器** | 超五类网线（经机巢→核心交换机） | MQTT | 汇总后的全部数据 | ≥100Mbps | N/A（有线） |

**3000m跑道能否覆盖？**

无人机图传距离标称≥5km（排弹方案数据），实际在开阔机场环境下3km完全在视距内，信号无遮挡。不需要中继。

**两架无人机同时飞会不会干扰？**

会。解决方案：
- 两机使用不同5.8GHz信道（信道间隔≥40MHz）
- 或将两机的地面站分置跑道两端，物理隔离

#### 4.2.3 探测载车 ↔ 服务器

| 项目 | 详情 |
|------|------|
| 主链路 | 4G/5G移动网络 |
| 协议 | MQTT |
| 数据内容 | 上传: GPR扫描数据(SEG-Y文件) + 磁力数据(时间戳+场值+GPS)；接收: 服务器下发的可疑目标坐标、处置任务 |
| 带宽 | ≥5Mbps |
| 延迟 | ≤500ms（非实时控制，可接受） |
| 备份 | 车载WiFi（回到机巢附近时自动连接局域网高速同步） |

**为什么不给载车装5.8GHz网桥？**

载车需要全场移动，但定向网桥需要对准。如果用全向天线，3km距离不够稳定。4G/5G覆盖最实际——机场都有基站覆盖。如果担心战时基站损毁，可在塔台架设备用小基站。

#### 4.2.4 排爆机器人 ↔ 服务器（关键设计）

这是整个系统通信设计中挑战最大的一段——机器人要进入弹坑、废墟、建筑背后，典型非视距（NLOS）场景。

**三层通信方案**：

| 层级 | 链路 | 使用场景 | 距离 | 延迟 | 可靠性 |
|------|------|---------|------|------|--------|
| **第1层：Mesh中继** | 自组网mesh网络（主用） | 机器人在跑道区域作业 | ≥3km（多跳） | ≤100ms | 高（自愈） |
| **第2层：4G/5G** | 移动网络（备用） | 超出mesh覆盖或mesh故障 | 不限 | ≤500ms | 中（依赖基站） |
| **第3层：有线光纤** | 光纤缆（精操作） | 机器人在10-100m内精细拆弹 | ≤100m | ≤5ms | 最高 |

**Mesh中继节点怎么部署？**

```
     塔台（根节点）
         │
    ┌────┴────┐
    │ 中继节点1 │──── 部署于跑道北端（固定杆/三脚架）
    └────┬────┘
         │
    ┌────┴────┐
    │ 中继节点2 │──── 部署于跑道中段（移动式，放工具箱里）
    └────┬────┘
         │
    ┌────┴────┐
    │ 排爆    │──── 也可作为mesh节点转发
    │ 机器人  │
    └─────────┘
```

- 中继节点本质是小盒子（≈路由器大小），内置电池（≥8小时），IP67防水
- 部署方式：作业前人工放到跑道两侧关键位置，作业后回收
- 节点数：最少2个，覆盖3km
- 协议：Babel mesh（USAF MDCR验证过的方案）或 OLSR
- 频段：900MHz（穿障能力强，适合NLOS）或 2.4GHz
- 每个节点吞吐量：≥50Mbps

**为什么要有线光纤精操作？**

排弹方案提到机器人"有线控制≥100m"。这是排爆作业的标准配置——当机器人距离弹药只有几米，在做精细拆解时，最大的风险是无线干扰/延迟导致误操作。有线光纤零延迟、零干扰、无限带宽。

机器人尾部拖一根轻质野战光纤（直径≈3mm，重量≈5kg/km），操作员在100m外有线操控。日常行驶用无线Mesh，精细作业切换有线。

### 4.3 通信中断怎么办？（离线容灾）

排弹方案明确要求"通信中断预案调用"。设计如下：

| 场景 | 无人机 | 机器人 | 载车 |
|------|--------|--------|------|
| 全系统断网 | 继续执行当前航线扫描，数据存机载存储。恢复后回传 | 继续执行当前任务（本机有弹药知识库+预设处置流程） | 数据存工控机本地 |
| 仅公网(4G/5G)中断 | 不受影响（用专有图传） | 切到Mesh中继 | 开回机巢附近接局域网 |
| 仅Mesh中断 | N/A（无人机不用Mesh） | 自动切4G/5G，如也断则执行当前任务后返航 | N/A |
| 服务器宕机 | 地面站缓存数据，服务器恢复后同步 | 机器人本机自主决策（简化版） | 工控机本地工作 |

    └─────────────────────────────────────┘
```

- 每机一次飞行覆盖半场，总任务耗时约1小时
- 如果电量不足以一趟飞完（标称续航53-59分钟），可在中间设置换电池点
- 如果只需快速普查（速度提到15m/s，降低重叠到20%），时间可压缩到35分钟

| **服务器选迎达RS6228H V5** | 排弹方案已指定型号，且Xeon 4210R 10核满足并发处理需求 |
