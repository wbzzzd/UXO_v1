## 背景

用户质疑 DDR-004 的两个问题：(1) 整体架构变化是否过大；(2) 修改后会不会依然出现冲突。架构师基于实际代码阅读（不是假设分析）做了二次验证，发现 DDR-004 存在 4 个设计缺口。本 DDR 记录验证结论和对 DDR-004 的修正。

---

## 验证结论

### 1. 架构变化规模评估（基于实际代码统计）

**现有代码**：6K 行（4464 行 .cpp + 1521 行 .h），42 个文件

| 类别 | 占现有代码比例 | 说明 |
|------|:---:|------|
| 保留不变 | ~75% | Widget 实现、3D 场景、Common、AirportData |
| 修改 | ~20% | MainWindow 重构(407→190行)、9 个头文件 include 路径、8 个 Widget 接口加 IDataProvider、CMake 调整 |
| 删除 | ~5% | loadMockData() 逻辑、部分 MainWindow 中间代码 |

**结论：不是"大重构"，而是"接口层调整 + 大量新增"。** 现有业务逻辑代码大部分保留。

### 2. 修改后的冲突点

| | 修改前 | 修改后 |
|---|---|---|
| 确定冲突 | 5 个 | 0 个（批次内） |
| 新概率性风险 | — | 4 个 |
| 净冲突 | 5 个确定 | 2-3 个概率性 |

修改后的 4 个新风险：

| 风险 | 概率 | 影响 | 说明 |
|------|:---:|:---:|------|
| include/core/ 类型冻结 vs 实际需求 | 中高 | 高 | Agent 发现缺字段需停工报告，可能退化为串行 |
| Batch 3 装配接口不匹配 | 40-60% | 中 | 函数签名/信号名/路径偏差，需 1-2 天修复 |
| CMake add_subdirectory 对不存在目录 | 低 | 低 | 用占位目录解决 |
| Widget 接口改动一致性 | 中 | 中 | 8 个 Widget 都要加 IDataProvider，遗漏导致装配失败 |

### 3. 是否值得做？

**关键发现：DDR-004 的墙钟时间不一定比 DDR-003 短。**

| 方案 | 估算 | 说明 |
|------|:---:|------|
| DDR-003 单 Agent 串行 | 12 天 | 三线有交叉依赖 |
| DDR-003 多 Agent 但不改架构 | 8-10 天 | 5 个冲突点导致频繁 merge 冲突 |
| DDR-004 改架构后多 Agent | 10-14 天 | Batch 0(2天) + Batch 1-4(8-12天) |

**但 DDR-004 的真正价值不在墙钟时间，而在于：**
1. **可预测性**：把"必然冲突"变成"概率风险"
2. **AI Agent 友好性**：AI Agent 无法可靠处理 C++ merge 冲突，消除冲突的价值远大于 2 天的 Batch 0 成本
3. **架构质量**：UiComponentRegistry、IDataProvider、模块化 CMake 是工业界标准实践

**诚实结论**：
- 如果是人类开发者：净收益可能持平
- 如果是 AI Agent：**值得做**，因为消除了 merge 需求

## DDR-004 的 4 个设计缺口及修正

### 缺口 1：createConnections 的去向未明确

**问题**：MainWindow.cpp L253-285 的 `createConnections()` 连接了 6 组跨 Widget 信号。DDR-004 说"拆分 MainWindow"但没说这部分怎么办。

**修正**：采用方案 C —— **通过 FrontendDataStore 信号中转**。
- Widget 连接 Store 信号而非互相连接
- 如 LeftPanelWidget::targetSelected 信号改为 FrontendDataStore::targetSelected
- 这与改造项 4（FrontendDataStore）对齐，但意味着 Widget 接口改动量更大

### 缺口 2：include/core/ 类型硬冻结过于严格

**问题**：Agent 写代码时发现 TargetInfo 缺字段，按 DDR-004 规则需停工报告，可能退化为串行。

**修正**：改为 **"字段不可删改，但允许尾部追加新字段"**。
- C++ struct 尾部加字段不破坏二进制兼容
- Agent 如需加字段，可直接在 struct 尾部追加，不需停工
- 但必须更新契约文档中的类型定义

### 缺口 3：CMake add_subdirectory 对不存在目录报错

**问题**：Batch 0 写好所有 `add_subdirectory()`，但 Batch 1/2 才创建这些目录。

**修正**：Batch 0 **创建所有空目录 + 占位 CMakeLists.txt**（每个只有一行 `# placeholder`）。

### 缺口 4：Batch 3 装配 buffer 未预留

**问题**：装配阶段发现接口不匹配需回溯到 Batch 2，打破并行模型。

**修正**：Batch 3 计划中**预留 1 天 buffer** 用于回溯修复。

## 修正后的执行计划

```
Batch 0（1 Agent，2天）
  ├── 提取共享类型到 include/core/（改 9 个文件 include 路径）
  ├── 拆分 MainWindow.cpp（407→190行，移出 createMainLayout + loadMockData）
  ├── 新建 UiComponentRegistry.h/cpp
  ├── 创建所有空目录 + 占位 CMakeLists.txt
  ├── 编写所有契约文档
  └── 冻结 include/core/（允许尾部追加字段）

Batch 1（4 Agents 并行，3-4天）
  ├── Agent A: HAL 接口（include/hal/*.h）
  ├── Agent B: MqttService（src/mqtt/*.cpp/h）
  ├── Agent C: NavigationWidget（src/ui/navigation/*.cpp/h）
  └── Agent D: 数据录制回放（src/recording/*.cpp/h）

Batch 2（3 Agents 并行，3-4天）
  ├── Agent E: 仿真设备+SensorSim（src/sim/*.cpp/h）
  ├── Agent F: FrontendDataStore（src/data/*.cpp/h，含跨Widget信号中转）
  └── Agent G: DecisionView+LeftPanel（src/ui/decision/*.cpp/h，通过 IDataProvider 解耦）

Batch 3（1 Agent，2天，含 1 天 buffer）
  └── Agent H: Application 装配 + 接口修复

Batch 4（1 Agent，1-2天）
  └── Agent I: 端到端验证
```

**总估算：11-14 天**（比原估的 10-14 天多 1 天，因 Batch 3 buffer）

---

## 关键风险提示

1. **include/core/ 类型演进**是最大风险。即使允许尾部追加，多个 Agent 同时追加同一 struct 仍会冲突。缓解：Batch 0 预留 `QVariantMap extensions` 扩展字段，Agent 优先用 extensions 而非加字段。
2. **Batch 3 装配**有 40-60% 概率需要回溯修复。Buffer 可吸收大部分情况。
3. **Widget 接口改动**涉及 8 个 Widget，任一遗漏导致装配失败。缓解：契约文档明确列出所有需要改的 Widget。

---

## 对 DDR-004 的最终修正

1. createConnections 采用方案 C（FrontendDataStore 信号中转）
2. include/core/ 类型允许尾部追加字段（非硬冻结）
3. Batch 0 创建所有空目录 + 占位 CMakeLists.txt
4. Batch 3 预留 1 天 buffer
5. TargetInfo 等结构体预留 `QVariantMap extensions` 扩展字段
