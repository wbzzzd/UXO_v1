# DDR-002: 硬件缺位下的开发策略

| 字段 | 值 |
|------|-----|
| **状态** | 已确认 |
| **日期** | 2026-06-24 |
| **参与人** | 产品经理（许清楚）、架构师（高见远）、主理人（齐活林） |

---

## 背景

排弹抢修指挥系统的所有硬件设备（红外相机、激光雷达、探地雷达、磁力探测、无人机、排爆机器人）均尚未到位，实际对接时间未知。团队希望提前推进软件开发。需要明确：在硬件缺位前提下，哪些能做、哪些不能做、怎么做。

---

## 问题诊断

### 硬件依赖度分析

对 SRS 定义的 69 个子功能逐一分析：

| 依赖程度 | 数量 | 占比 | 说明 |
|---------|------|------|------|
| 零依赖（可独立开发） | 36 | 52% | 纯软件功能，如决策算法、GIS、数据库、3D 渲染 |
| 弱依赖（部分可行） | 14 | 20% | 算法框架可开发，参数需硬件标定 |
| 强依赖（必须等硬件） | 19 | 28% | 传感器控制、飞控闭环、排爆执行 |

**关键发现：72% 的功能可以在无硬件条件下推进。**

### 各模块硬件依赖度

| 模块 | 可独立开发比例 | 说明 |
|------|-------------|------|
| 信息标注 | 100% (9/9) | 硬件依赖最低 |
| 决策逻辑 | 100% (9/9) | **系统核心价值所在，却代码为零** |
| 插件模块 | 80% (8/10) | 仅取证/排除控制需硬件 |
| 态势管理 | 78% (7/9) | 仅实时视频叠加需硬件 |
| 目标分类 | 73% (8/11) | AI 模型可用公开数据集训练 |
| 全域探测 | 19% (3/16) | 大部分传感器控制需硬件 |
| 排爆处置 | 0% (0/5) | 全部强依赖硬件 |

### 仿真可行性边界

| 保真度 | 可模拟项 | 不可模拟项 |
|--------|---------|-----------|
| 高保真 | 目标位置/类型、设备状态、3D 场景、告警、气象、GPS 轨迹、通信延迟 | - |
| 中等保真 | 可见光/红外图像（公开数据集）、激光点云（KITTI）、GPR 信号（gprMax）、机械臂运动（Gazebo） | - |
| 低保真 | - | 引爆/殉爆效应、传感器噪声特性、飞控动力学、无线链路质量、机械臂力控精度 |

**关键风险：殉爆安全链算法涉及生命安全，模拟数据不可用于最终决策依据，必须经实弹试验验证。**

---

## 讨论过程

### 产品经理观点

硬件缺位不是开发停滞的理由。52% 的功能可以立即推进，尤其是决策逻辑模块——9 个子功能全部零硬件依赖，是系统的核心业务价值，却至今代码为零。当前最大的产品风险是"继续画 UI 而不写算法"。

MVP 应重新定义为"无硬件条件下可验证的最大功能集合"，核心是打通"模拟传感器 → MQTT → 存储 → 业务逻辑 → UI 显示"的端到端数据链路。

### 架构师观点

设计 HAL（硬件抽象层）解决"硬件不在场也能开发"的问题。核心思路：上层模块从"依赖具体硬件"变为"依赖抽象接口"（IDevice → ISensor/IActuator → ICamera/ILidar/IGpr/IDrone/IUGV）。仿真设备（SimCamera/SimDrone 等）实现这些接口，提供数据回放、故障注入、延迟模拟、噪声注入等能力。

当硬件到位时，只需实现 RealCamera/RealDrone 等，修改配置文件从 `"mode": "simulation"` 切换到 `"mode": "hardware"`，**上层代码一行不改**。

### 主理人判断

两位的方案互补：PM 定义了"做什么"（72% 功能可推进 + MVP 范围），架构师定义了"怎么做"（HAL 抽象层 + 仿真设备）。两者合在一起就是完整的硬件缺位开发策略。

---

## 决策

### 1. 采用 HAL + 仿真设备架构

建立硬件抽象层，所有硬件设备通过 IDevice/ISensor/IActuator 接口抽象。开发期间使用仿真设备实现，硬件到位后切换为真实设备实现。

### 2. MVP 范围定义

```
纳入 MVP（无硬件可验证）:
  ├── HAL 接口定义 + 仿真设备层
  ├── MQTT 通信层
  ├── SQLite 存储层
  ├── 决策逻辑模块（全部 9 个子功能）
  ├── 目标分类模块（8 个可独立开发子功能）
  ├── 信息标注模块（全部 9 个子功能）
  ├── 态势管理模块（7 个可独立开发子功能）
  ├── 插件模块（8 个可独立开发子功能）
  └── 前端 FrontendDataStore + UI 骨架补全

不纳入 MVP（等硬件）:
  ├── 全域探测模块（仅接口定义）
  ├── 排爆处置模块（仅接口定义）
  └── 实时视频叠加
```

### 3. 三线并行开发

| 线 | 内容 | 负责人 | 依赖 |
|----|------|--------|------|
| 线1 | HAL 接口 + 仿真设备层 | 架构师 + 工程师 | 无 |
| 线2 | FrontendDataStore + UI 骨架补全 | 工程师 | 无 |
| 线3 | MQTT 通信层 + SensorSim | 工程师 | 无 |
| 汇合 | Application::initializeModules() 装配 | 主理人协调 | 三线各自跑通 |

### 4. 殉爆安全链算法特殊处理

殉爆安全链计算可在无硬件条件下开发算法，但必须：
- 在系统中标注为"未经验证"
- 设置极大的安全系数
- 争取在交付前完成至少 1-2 次实弹标定试验

---

## 理由

- HAL 架构的依赖倒置原则使上层模块不关心是真实还是仿真设备，这是"硬件缺位也能开发"的技术基础。
- 三线并行是因为三条线互不依赖（HAL 接口定义、前端 Store、MQTT 通信可独立开发），汇合点在 Application 装配，最大化并行效率。
- MVP 范围基于 72% 功能可独立开发的客观数据，不是主观判断。

## 风险清单

| 排名 | 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|------|---------|
| #1 | 殉爆安全链算法未经实弹验证 | 中 | 致命 | 标注"未经验证" + 极大安全系数 + 争取实弹标定 |
