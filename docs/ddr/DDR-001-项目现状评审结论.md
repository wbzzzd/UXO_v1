## 背景

排弹抢修指挥系统（UXO Mission Control）当前处于早期开发阶段。在制定下一步计划前，需要从产品和架构两个视角对现有代码进行独立评审，判断当前代码是否合理、是否应该继续、如何继续。

## 问题诊断

### 1. 当前代码覆盖了 SRS 的多少？

**结论：约 10% 的 P0 需求，集中在 UI 壳子和数据结构定义。**

| SRS 模块 | 覆盖程度 | 说明 |
|---------|---------|------|
| 全域探测 | ~5% | 仅有起降按钮 UI 占位，无传感器控制 |
| 目标分类 | 0% | Classification/ 目录为空 |
| 信息标注 | ~40% | 3D 场景渲染完成，但缺少标注编辑、图层管理 |
| 态势管理 | ~25% | 3D 场景 + 目标列表存在，但无真实数据融合 |
| 决策逻辑 | ~10% | DecisionSuggestionPanel 有 UI 占位，无任务规划/路径规划/殉爆安全链 |
| 排爆处置 | 0% | Disposal/ 目录为空 |
| 插件模块 | 0% | plugins/ 目录为空 |

### 2. 架构层面有什么问题？

**P0 阻断级（必须先解决才能继续开发）：**

| ID | 问题 | 具体位置 | 影响 |
|----|------|---------|------|
| P0-1 | 五大业务模块完全缺失 | `include/Core/Detection/` 等目录下仅头文件声明，`.cpp` 均为空 | 排弹指挥核心流程不存在 |
| P0-2 | 3D 场景不支持增量更新 | `AirportSceneFactory` 只有 `createScene()`，无 `addTarget()`/`updateTarget()`；`SituationView::loadAirportData()` 删除整个根实体再重建 | 炸弹标注无法实时变化（系统核心功能） |
| P0-3 | Application 初始化全部为 Stub | `Application.cpp` 中 `initializeDatabase()`、`initializeCommunication()` 等均直接 `return true` | 系统无法连接任何真实后端 |

**P1 重要级：**

| ID | 问题 | 具体位置 | 影响 |
|----|------|---------|------|
| P1-1 | 无数据抽象层 | `MockDataGenerator` 直接被 `MainWindow::loadMockData()` 调用，数据直接 push 到 Widget 私有成员 | 真实数据接入需大规模改动 UI 代码 |
| P1-2 | Common → Core 隐式反向依赖 | `MockDataGenerator.h` 引用 `Core/Data/Types.h`，但 CMake 中 Common 未链接 Core | 分层架构被破坏 |
| P1-4 | MainWindow God Object | `MainWindow.cpp` ~400 行，承担布局/信号连接/数据加载/菜单创建全部职责 | 扩展性极差 |
| P1-5 | TargetMarker 管理无索引 | `SituationView.cpp` 中通过 `objectName()` 字符串线性搜索 | O(n) 查找，目标增多后性能劣化 |
| P1-6 | QVector3D 精度绑定 | `Types.h` 全局使用 `QVector3D`（float 精度） | 军用场景下精度可能不足；Core 层依赖 Qt GUI |

### 3. 产品层面有什么问题？

- **"展厅式开发"**：UI 层做到 90% 完整度，但核心业务逻辑 0%。看起来像完整系统，实际不可运行、不可验证。
- **DecisionView 是空文件**：构造函数只初始化 nullptr，连 `setupUi` 都没调用。
- **NavigationWidget 点击只 qDebug**：6 个导航项点击后无视图切换，只输出日志。
- **数据流是"推一次就结束"**：`loadMockData()` 一次性生成所有数据，无法模拟持续探测发现新目标。

---

## 讨论过程

### 产品经理观点

当前把 UI 层做到 90% 完整度但核心业务逻辑 0%，是典型的"展厅式开发"。这种做法在军工项目中风险尤其高——无法早期发现技术难题（如 3D 渲染在国产 OS 上的性能），给干系人错误信心。建议立即冻结纯 UI 开发，将开发力量转移到通信层和决策算法。

### 架构师观点

当前代码**不需要推倒重来**。分层骨架方向正确，GlobalStyle 样式体系专业完整，enum class 枚举设计合理，AirportSceneFactory 的数据驱动渲染模式方向对。但需要先花 1-2 天完成三项关键重构（数据抽象层、标记管理器、分层修正），否则技术债务将以指数级增长。

### 主理人判断

两位一致认为可以继续但需调整方向。架构师的三项重构和 PM 的"冻结 UI、转向业务逻辑"不矛盾——先修架构债再铺功能。技术方案上，SQLite 先行（务实）+ 接口抽象预留（严谨），两者兼容。

---

## 决策

1. **继续开发，不推倒重来**。当前代码的基础架构骨架（四层 CMake 分层、enum class 枚举、GlobalStyle 样式、AirportSceneFactory 数据驱动渲染）保留。
2. **冻结纯 UI 开发**。不再新增 UI 页面或美化，直到通信层和数据抽象层就位。
3. **优先完成三项架构重构**：
   - (a) 建立数据抽象层（IDataProvider 接口 + FrontendDataStore）
   - (b) 重构 3D 场景标记管理（TargetMarkerManager + 增量更新）
   - (c) 修正分层依赖（MockDataGenerator 移出 Common）
4. **随后进入端到端闭环建设**：MQTT 通信层 + SQLite 存储 + DetectionModule 最小可用。

---

## 理由

- 不重构直接写功能代码，PM 说的"大概率返工"风险极高。三项重构解决了 3 个 P0 和 2 个 P1 问题，工作量约 1-2 天，但避免了后续指数级的技术债务。
- 冻结 UI 是因为当前 UI 的交互模式高度依赖后端数据的能力边界（如路径规划算法能处理多少航点、传感器数据刷新率是多少），没有后端 constraint 的前端设计大概率返工。
