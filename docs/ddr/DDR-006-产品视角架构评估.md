## 背景

用户质疑："这个修改是不是只为了开发服务？我们最终还是在做产品的，我更关注架构是否贴合产品。"

架构师重新阅读 SRS 和 SDD，从产品视角全面评估当前架构。结论：**DDR-001~005 解决的全是"怎么把代码写出来"的问题，但没有回答"这个产品的架构应该长什么样"。**

---

## 产品本质

排弹抢修指挥系统的核心价值不是"探测+识别+处置"的功能罗列，而是：**在敌方打击后，用最少的人力暴露、在最短时间内恢复机场跑道可用性。**

其中"安全"是硬约束——殉爆可导致连锁爆炸，摧毁油库弹药库，让整个场站瘫痪。任何架构决策如果牺牲安全性换取开发便利，就是本末倒置。

### 关键质量属性（按优先级）

| 优先级 | 质量属性 | SRS 依据 |
|--------|---------|---------|
| P0 | 安全性 | 6.2 功能安全：殉爆防护、紧急停止、误操作防护 |
| P0 | 实时性 | 5.1：点云≥100万点/s、识别≤500ms、融合≤1s、渲染≥30fps |
| P1 | 可靠性 | 5.3/7.3：7×24运行、MTBF≥10000h、MTTR≤30min |
| P1 | 离线能力 | 5.3：离线工作≥2小时；F-091：通信中断自动调用离线预案 |
| P2 | 信息安全 | 6.1：认证、权限、加密、审计 |
| P2 | 可维护性 | 约束条件：麒麟/统信 |

## 架构缺失盲区

### 盲区 1：安全架构完全空白（致命）

系统控制物理机器人处理爆炸物，但架构层面：
- **没有安全边界设计**——设备控制指令从决策模块直接到 HAL，中间无安全校验层
- **紧急停止没有架构位置**——SRS 要求"一键停止所有设备"，但所有 DDR 中完全没提
- **没有 fail-safe 设计**——系统在处置过程中崩溃，机器人应该回到什么状态？
- **殉爆安全链没有独立性**——只是 PathPlanner 的一个方法，其他模块的 bug 可能影响安全链计算

### 盲区 2：实时数据管线有根本性缺陷（严重）

数据量计算：
- 一个点 = x+y+z+intensity ≈ 16 字节
- 100万点/s × 16字节 = 16MB/s/传感器
- 8路传感器 = 128MB/s

**DDR-003 设计的 MQTT Topic `uxo/sensor/{deviceId}/data` 无法支撑 128MB/s 点云数据。MQTT 不是干这个的。**

SRS 自己的接口设计（I-002）明确写了"数据处理接口：共享内存"，但 DDR-003 完全忽略。

### 盲区 3：离线模式无设计（严重）

SRS 要求通信中断后持续工作≥2小时。需要回答：
- 传感器数据离线时怎么缓存？满了怎么办？
- 设备控制指令离线时怎么排队？恢复后怎么补发？
- 离线预案（F-091）什么时候触发？由谁触发？

当前架构没有任何模块负责这些问题。

### 盲区 4：可靠性架构缺失（严重）

SRS 要求 MTBF≥10000 小时、7×24 不间断运行。当前架构：
- `Application::run()` 是空函数
- 无进程守护、无健康监控、无崩溃恢复、无内存泄漏防护、无日志轮转

这是桌面应用架构，不是 7×24 系统架构。

### 盲区 5：殉爆安全链无架构隔离（致命）

殉爆安全链是产品最核心的安全功能，但：
- 在 SDD 中只是 `PathPlanner::calculateSafeDistance()` 的一个方法
- 与其他路径规划代码混在一起
- 无独立性保障

## 架构决策与产品需求的矛盾

| 矛盾 | 问题 | 影响 |
|------|------|------|
| MQTT 作为唯一数据通道 | 无法支撑 128MB/s 高带宽实时数据 | 性能指标全部不可能达标 |
| FrontendDataStore 进程内单例 | 无法跨终端共享状态 | 多席位协同（≥10人）不可能实现 |

## DDR-004/005 修改的产品价值评估

**对产品有长期价值的（保留）**：
- ✅ HAL 抽象层——产品需要硬件独立性
- ✅ DataStore 模式——产品需要数据一致性
- ✅ 仿真设备层——产品需要训练和测试能力
- ✅ 数据录制回放——产品需要事后分析能力

**纯开发便利但无害的（保留，不过度投入）**：
- ⚠️ UiComponentRegistry——为多 Agent 并行设计，多一层间接调用
- ⚠️ IDataProvider 接口——为解耦设计，有关注点分离价值

**需要警惕的**：
- ⚠️ QVariantMap extensions 是类型安全反模式，不要用
- ⚠️ Batch 0 类型冻结是开发约束不是产品决策，过早冻结可能导致领域建模不准确

## 决策：Phase 1 新增 5 项架构工作

### 新增 A：安全架构骨架（P0，必须）

| 组件 | 职责 |
|------|------|
| `SafetyManager` | 全局安全状态机（Normal → Alert → Emergency → Recovery） |
| `EmergencyStopController` | 紧急停止控制器，独立于所有业务模块，最高优先级中断所有操作 |
| `SafetyInterlock` | 安全联锁层——所有设备控制指令经安全校验后才下发 |
| `FailSafeHandler` | 故障安全处理——系统崩溃时设备回到安全状态 |

Phase 1 目标：EmergencyStopController 能一键停止所有仿真设备，SafetyInterlock 能拦截不安全指令。

### 新增 B：分层数据通道（P0，必须）

| 通道 | 传输内容 | 技术 |
|------|---------|------|
| 高带宽数据通道 | 点云、图像流、雷达信号 | 共享内存或 ZeroMQ PUB/SUB |
| 控制遥测通道 | 设备状态、位置、电量、控制指令 | MQTT |
| UI 更新通道 | 态势刷新、目标列表 | Qt 信号槽 |

Phase 1 目标：架构上必须有分层——SimCamera 的点云数据走共享内存不走 MQTT。

### 新增 C：离线模式骨架（P1，应该）

| 组件 | 职责 |
|------|------|
| `OfflineManager` | 检测通信状态，切换在线/离线模式 |
| `OfflineDataCache` | 离线时缓存传感器数据，恢复后同步 |
| `CommandQueue` | 离线时排队控制指令，恢复后补发 |

Phase 1 目标：OfflineManager 能检测 MQTT 断连，切换离线模式，SimDrone 仍能执行本地预编程航线。

### 新增 D：可靠性骨架（P1，应该）

| 组件 | 职责 |
|------|------|
| `HealthMonitor` | 定期检查各模块健康状态（心跳+资源占用） |
| `Watchdog` | 主进程守护，检测模块崩溃，自动重启 |
| `LogManager` | 日志轮转，防止磁盘写满 |

Phase 1 目标：HealthMonitor 检测 MQTT 连接状态、数据库连接状态、仿真设备存活状态。

### 新增 E：殉爆安全链独立模块（P1，预研）

- 从 PathPlanner 中独立为 `SafetyChainModule`
- 独立编译、独立测试、标注"未经验证"
- Phase 1 建立模块骨架和安全距离计算 stub

## 对 DDR-003 Phase 1 执行计划的影响

原 DDR-003 的 10 个改造项全是开发基础设施。本决策新增 5 项架构工作，使 Phase 1 从"开发脚手架"升级为"产品架构骨架"。

修订后的 Phase 1 = 原 10 改造项 + 新增 5 项 = 15 项。

其中新增 A（安全架构）和新增 B（分层数据通道）是 P0，必须与原改造项同等优先级执行。新增 C/D/E 是 P1，可在原改造项完成后跟进。

## 诚实承认

架构师承认在 DDR-001~005 过程中有一个明显的产品视角盲区：**把"架构"等同于了"代码结构"**。

DDR-001 评审代码分层，DDR-002 设计硬件抽象层，DDR-003 规划文件改动，DDR-004/005 解决 AI Agent 并行——这些都是"代码架构"，不是"产品架构"。

产品架构要回答的是：安全怎么保证？性能怎么达标？7×24 怎么实现？离线怎么工作？这些问题的答案是**运行时行为设计**——数据怎么流、故障怎么处理、安全怎么兜底。

---

## 核心判断

当前架构是一个优秀的"开发脚手架"，但不是一个"产品架构"。Phase 1 必须在开发基础设施之外，补上安全架构和分层数据通道这两个 P0 缺口，否则即使代码全部写完，这个系统也不敢用于真实排弹场景。
