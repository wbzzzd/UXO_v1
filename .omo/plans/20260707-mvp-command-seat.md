## 背景

- 当前项目是 Qt 5 / CMake / C++17 桌面客户端，目标程序名为 `UXOMissionControl`。
- `MainWindow`、`SituationView`、`DeviceControlView`、`DecisionView`、`StatusBarWidget` 已有初步代码。
- 阶段 1 前，`src/App/main.cpp` 只创建 `App::Application` 并进入 `app.exec()`；阶段 1 已改为初始化成功后调用 `app.run()`，由 `Application::run()` 创建并显示 `MainWindow`。
- `Application::initialize()` 当前是占位初始化，配置、日志、数据库、通信、模块初始化函数都直接返回成功。
- `config/devices.json` 和 `config/system.json` 存在，但包含未来真实通信字段；MVP 阶段只可按模拟/只读数据使用。

## 目标

做出一个可演示、可理解、风险可控的指挥席 MVP 闭环：启动后可见主窗口，展示模拟任务、目标、设备状态、处置方案，并支持最小的模拟流程状态变化。

## 非目标

- 不接真实设备。
- 不发送真实排弹、机器人、无人机、传感器或外部通信命令。
- 不写入真实数据库。
- 不实现安全关键自动决策。
- 不一次性补齐 SRS/SDD 中的完整目标架构。

## 场景契约

### 场景 1：启动可见主窗口

- 类型：happy path。
- 验收条件：运行 `UXOMissionControl` 后出现标题为“排弹抢修指挥系统 V1.0”的主窗口，并包含菜单栏、工具栏、状态栏、左侧 Dock、右侧 Dock 和中心区域。
- 证明表面：桌面 GUI 截图或可用的 Qt 自动化/最小启动测试。
- 测试要求：已添加 `startup_visible` 验证点，证明 `Application::run()` 会显示 `MainWindow`。

### 场景 2：模拟设备与任务数据展示

- 类型：happy path。
- 验收条件：UI 能展示至少 1 个模拟任务、1 个模拟目标、2 类模拟设备状态，并明确标注“模拟”。
- 证明表面：运行应用后截图，或可替代的 UI 组件状态验证输出。
- 测试要求：先为模拟数据加载/转换写测试，再接入 UI。

### 场景 3：空数据或配置缺失

- 类型：edge。
- 验收条件：缺少模拟任务或设备数据时，应用仍能启动并显示明确的空状态，不崩溃、不尝试真实连接。
- 证明表面：测试输出和运行表面截图/日志。
- 测试要求：先构造空数据测试，确认失败后实现最小处理。

### 场景 4：安全边界回归

- 类型：adjacent-surface regression。
- 验收条件：MVP 过程中不引入真实设备连接、外部控制命令、数据库写入或明文密钥。
- 证明表面：代码搜索、配置检查和安全 review 输出。
- 测试要求：每阶段完成后运行安全检查命令或等价搜索。
